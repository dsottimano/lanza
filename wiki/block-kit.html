Block Kit & the form-block fix

Block Kit is the declarative JSON UI that sandboxed plugins render with — the host draws the blocks, no plugin JavaScript runs in the browser.

How it works

  1. User opens a plugin admin page → admin sends a page_load interaction to the plugin's admin route.
  2. Plugin returns { blocks: [...] }.
  3. Admin renders them with BlockRenderer (from @emdash-cms/blocks).
  4. User interacts (button click / form submit) → interaction posts back → plugin returns new blocks.

Available blocks & elements

Blocks: header, section, divider, fields, table, actions, stats, form, image, context, columns, chart, code, meter, banner.

Form elements: button, text_input, number_input, select, toggle, secret_input, checkbox, radio, date_input, combobox.

Keys are snake_case throughout: action_id, block_id, initial_value.

Case study: fields is not iterable

The demo plugin's admin page crashed with this exact error. Root cause, traced to the real renderer in @emdash-cms/blocks/dist/index.js:

function getInitialValues(fields) {
  for (const field of fields) /* ... */     // ← throws if `fields` is undefined
}
function FormBlockComponent({ block, onAction }) {
  const [values] = useState(() => getInitialValues(block.fields));
  // ...
}

A form block's inputs must live under fields. The demo passed them under elements, so block.fields was undefined → "fields is not iterable."

The fix

Wrong (crashes)Correct
elements: [ ... ]fields: [ ... ]
blockId: "add-note"block_id: "add-note"
 
{
  type: "form",
  block_id: "add-note",
  fields: [
    { type: "text_input", action_id: "note_text", label: "New note" }
  ],
  submit: { label: "Add note", action_id: "add_note" }
}
Why TypeScript didn't catch it: the blocks array is typed any[] in the handler, so tsc can't flag a wrong key name — only the runtime renderer does. Always reload the actual page to verify Block Kit changes.
Dev-server gotcha that cost us time: a plugin's sandbox-entry.ts is imported once at dev-server startup. A browser hard-refresh reloads frontend assets but not the cached server-side plugin module. After editing plugin code, restart npx emdash dev — otherwise you keep seeing the old behavior.
