Plugin capabilities & limits

What an EmDash plugin can do to the admin — and the hard boundaries it can't cross.

Two plugin formats

FormatAdmin UIWhere it runsMarketplace?
Standard (default)Block Kit (JSON blocks)Sandboxed isolate on Cloudflare, in-process elsewhereYes
Native (escape hatch)React or Block KitAlways in the host process (trusted)No

Your site's plugins are declared in astro.config.mjs → trusted mode, so native (React) admin UIs are on the table if you need them.

What a plugin CAN do

  • Add admin pages. adminPages: [{ path, label, icon }] — each becomes a sidebar item routed to /plugins/<id><path>.
  • Add dashboard widgets via adminWidgets (Block Kit cards).
  • Store data. Plugin-private document collections (auto-created, indexed queries) + a KV store. No capability needed; scoped to the plugin.
  • React to events. Hooks like content:afterSave, plugin:install, email hooks, etc.
  • Expose API routes at /_emdash/api/plugins/<id>/<route>.
  • Read/write real CMS content — gated by declared capabilities (content:read, content:write, media:*, users:read, network:request, email:send, …).
  • Ship a full React admin page — native format only.

What a plugin CANNOT do

A plugin cannot modify the default menu in any way. It can only append its own items.

Verified in buildNavItems — the core nav is a hardcoded array; plugin pages are pushed onto the end, namespaced so they can't even collide with a core route:

for (const [pluginId, config] of Object.entries(manifest.plugins)) {
  if (config.enabled === false) continue;
  for (const page of config.adminPages)
    items.push({
      id: `plugin-${pluginId}-${page.path}`,
      title: label,
      to: `/plugins/${pluginId}${page.path}`,
      icon: PuzzlePiece,        // ← see gotcha below
      keywords: ["plugin", pluginId],
    });
}

So a plugin can't rename, remove, reorder, or replace any default item.

Icon gotcha: the plugin's declared icon (e.g. the demo's "sparkles") is ignored in the sidebar. Notice icon: PuzzlePiece is hardcoded above — every plugin page shows the puzzle-piece icon in this version.

The demo plugin, as a reference

Earlier revisions of this starter shipped a demo plugin that demonstrated the whole plugin surface: a private notes storage collection, a content:afterSave hook that logs a note on every save, a read of real post counts via content:read, and a Block Kit admin page. It has been removed — per the starter's Rule 3 (standard Workers, no Dynamic-Worker / sandbox plugins) the production build ships only first-party + the local email provider, and the demo carried a sandbox-entry.ts that had no place in a hardened starter. The pattern above still documents what a plugin can do; build your own in emdash/packages/plugins/ following the email-resend plugin (written format: "native", in-process — no sandbox).
