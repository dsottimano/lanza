Per-entry page width

Every entry can choose its own page width / margin from the admin — full screen, wide, the default reading column, narrow, or a custom side-padding percentage. It's a content-type feature: two fields on the collection plus a shared renderer helper. No EmDash core edits, no extra D1/R2 reads.

The one idea: layout can be content. A select field on the collection becomes a width dropdown in the admin; the renderer maps the chosen value to a CSS width. Because the value rides along in the entry data that's already in the cached render, the control costs zero extra database hits.

The two fields

Add these to any collection's fields array in seed/seed.json (they're already on posts and pages):

{ "slug": "layout_width", "label": "Page width", "type": "select",
  "options": [
    { "value": "default", "label": "Default reading column" },
    { "value": "wide",    "label": "Wide" },
    { "value": "full",    "label": "Full screen" },
    { "value": "narrow",  "label": "Narrow" } ] },
{ "slug": "layout_padding", "label": "Custom side padding (%)", "type": "number" }

layout_width renders as a dropdown; layout_padding as a number input. A custom padding value overrides the preset — it means "full-bleed, inset by this much on each side" (your 20%-margin case), clamped to 45% so an editor can't collapse the column to nothing.

How it renders

One shared helper does the mapping — src/utils/layout-width.ts:

const layout = resolveLayout(entry.data);
// → { "data-width": "wide" }                         (preset)
// → { "data-width": "full", style: "--page-pad-x: 20%" }  (custom padding)

Renderers spread that onto the article wrapper (<article ...{...layout}>). Scoped CSS keys off the data-width attribute and reads the optional --page-pad-x custom property:

.page-article { max-width: var(--page-max-width, var(--content-width));
                padding-inline: var(--page-pad-x, var(--spacing-6)); }
.page-article[data-width="wide"]   { --page-max-width: var(--wide-width); }
.page-article[data-width="narrow"] { --page-max-width: 36rem; }
.page-article[data-width="full"]   { --page-max-width: none; }

It applies to every content type — including new ones

Collections served by the pattern-routed catch-all (src/pages/[...path].astro) consume resolveLayout already, so a content type you add later inherits the control for free — just paste the two fields into its schema. Only the bespoke posts/[slug].astro template wires it by hand, and deliberately differently:

Posts keep their gutters. On a post, the width setting grows the centre reading column (the grid's body track) rather than flattening the three-column editorial layout — the author/meta column and the TOC gutter stay. "Full" lets the body fill the remaining space; it does not delete the gutters. (See Customizing the admin for why that layout is intentional.)

What each setting looks like

Verified on the running app against a seeded post:

SettingResult
default680px centred reading column, left meta + right TOC gutters (the baseline).
wideBody column grows to ~900px; gutters retained.
fullBody fills the container edge-to-edge, hero full-bleed; gutters retained (not flattened).
narrowTight ~576px column with more surrounding whitespace.
layout_padding = 20Full width inset 20% on each side (overrides the preset; clamped at 45%).

Turning it on

The two fields are columns on the collection's table, so they need a schema migration before they appear in the admin:

npx emdash dev      # runs migrations — adds layout_width / layout_padding
Safe before you migrate. If the columns don't exist yet, entry.data.layout_width is undefined, resolveLayout returns default, and pages render exactly as before — no errors. The migration is additive (it only adds the two columns), so it won't touch existing content.

Why it's rule-compliant

  • No core edits — fields live in seed.json, logic in src/utils/ + the page templates.
  • No extra D1/R2 reads — the values are part of the entry already fetched and edge-cached.
  • Cache-safe — width is baked into the cached HTML and purges with the entry's normal cache tag on publish.
