Pattern-driven routing

How we made the admin URL Pattern field actually control public URLs — using EmDash's own resolveEmDashPath resolver in a catch-all route. No core EmDash code was modified.

Context: by default the URL Pattern field is a footgun — it generates link strings but does not route anything (see Admin menu architecture). This page documents the change that closes that gap.

The problem it solves

Out of the box, two independent systems decide URLs:

SystemOwned byControls
Actual routesAstro files in src/pages/What responds to a request (e.g. pages/[slug].astro → /pages/{slug})
URL Pattern (urlPattern)The admin DB recordOnly string generation: preview "View" link, sitemap.xml, slug-rename auto-redirects, canonical links

So setting URL Pattern to /{slug} made the admin link to /about, but no Astro file served the root /{slug} → 404. The field looked like it controlled routing; it didn't.

The fix: a catch-all that reads the pattern at request time

EmDash exports a supported resolver, resolveEmDashPath(path), which (per its own docs) "loads all collections with a urlPattern set, converts each pattern to a regex, and tests the given path. On match, extracts the slug and fetches the entry."

We added one Astro file — a rest/catch-all route — that calls it. This is the missing piece the template didn't ship:

// emdash/src/pages/[...path].astro   ← YOUR repo, not node_modules
import { resolveEmDashPath, getEmDashEntry } from "emdash";
import { PortableText } from "emdash/ui";
import Base from "../layouts/Base.astro";

const path = `/${Astro.params.path ?? ""}`;

// Ask EmDash which collection + entry this path maps to,
// using the admin-configured URL patterns.
const resolved = await resolveEmDashPath(path);
if (!resolved) return Astro.redirect("/404");

const { collection, params } = resolved;
// Re-fetch for the cache hint + visual-editor edit proxy.
const { entry, cacheHint } = await getEmDashEntry(collection, params.slug);
if (!entry) return Astro.redirect("/404");
if (Astro.cache?.enabled) Astro.cache.set(cacheHint);
// → render entry.data.title + <PortableText value={entry.data.content} />

Now whatever you type in the admin field controls the live URL. Set /{slug} → /about works. Set /help/{slug} → /help/about works. Zero per-page code. We also deleted the now-redundant src/pages/pages/[slug].astro (the catch-all is strictly more general — it serves /pages/{slug} too if you ever switch the pattern back).

Route priority — why this is safe

Astro resolves specific routes before the rest route. The catch-all ([...path]) has the lowest priority, so it only fires for paths nothing else matches:

  • Wins first: /, /posts, /posts/[slug], /category/[slug], /tag/[slug], /search — untouched.
  • Catch-all handles: pages and any new custom collection whose pattern points outside those reserved trees.
Scope limit: posts, category, and tag still use their dedicated file routes (which correctly take priority). This catch-all governs pages and new collections — not posts. Making posts pattern-driven too would be a separate change.

Is this modifying core EmDash? No.

The catch-all is a normal first-party Astro page in your repo. It only imports a function EmDash already exports publicly (from "emdash") — the supported integration point. Nothing under node_modules/emdash is touched.

How to use it

  1. In admin: Content Types → (collection) → URL Pattern, set e.g. /{slug}, Save.
  2. That's it — the catch-all reads the new pattern at request time. No code change, no redeploy of route files.

Patterns are matched against the request path with {slug} (and {id}) as placeholders, e.g. /{slug}, /blog/{slug}, /help/{slug}.

Gotcha: deleting a route under a running dev server

After deleting pages/[slug].astro, the dev server threw:
No cached compile metadata found for "…/pages/[slug].astro?astro&type=style…"

Cause: a browser tab still held HTML rendered before the deletion, which links to the deleted file's scoped stylesheet (…[slug].astro?astro&type=style…). Each refresh re-requested that dead module → Vite threw. Vite's on-disk caches (node_modules/.vite, .astro) can pin it too. It is not a code error.

Fix:

  1. Stop all astro dev processes.
  2. Clear caches: rm -rf node_modules/.vite .astro
  3. Start one fresh server: npm run dev
  4. Hard-refresh the browser (Cmd/Ctrl+Shift+R) or open a new tab — don't reload the stale tab. This is the step that actually clears it.
