Starter rules & live audit

This starter exists to correct EmDash and keep a Cloudflare deployment cheap and upgrade-proof. Five rules enforce that. They are also encoded in the repo's CLAUDE.md so any agent working here obeys them. Below: each rule, how it is enforced today, and the audit that brought the codebase into line.

The one idea: never fork EmDash. Every correction is a project-level hook — middleware, an injected route, config, or a local plugin — so pnpm update emdash never clobbers your fixes.

The five rules

  1. Never touch EmDash core. No edits to node_modules/emdash, @emdash-cms/*, or pnpm patches of their dist/. Work around core bugs with hooks (Rule 5).
  2. Optimize aggressively to limit D1 & R2 queries. Cache first, serve media straight from R2's public domain, use D1 read replicas (session: "auto"), coalesce writes.
  3. Only standard Workers — never Dynamic Workers, no third-party plugins. Plugins run in-process (plugins:, not sandboxed:). First-party (@emdash-cms/*) and local (@local/*) only.
  4. Leverage Cloudflare caching at all times. Every public route gets a routeRules entry and an Astro.cache.set(cacheHint) call so publishing purges by tag.
  5. Use hooks to modify EmDash. Astro middleware (onRequest) and integration hooks (astro:config:setup → injectRoute) are the sanctioned mechanism — they are how Rule 1 stays true.

Audit findings

✅ Rule 1 — fixed: the core patch is gone

The repo used to carry a pnpm patch that edited dist/astro/middleware.mjs inside the emdash package to add one missing binding — handleContentAuthors, which EmDash 0.19 forgot to wire, 500-ing the admin author-filter dropdown. That was a direct modification of core and broke Rule 1. It has been replaced by a hook (Rule 5): src/middleware.ts now runs a restoreContentAuthors middleware that grafts the handler back onto locals.emdash. It works because EmDash injects its middleware with order: "pre", so locals.emdash (including .db) is already populated when ours runs; the handler is just the public handleContentAuthors(db, collection) export, exactly as the runtime method calls it. It fills the slot only when ABSENT, so a future EmDash that binds it itself silently takes over. The patches/ dir and the patchedDependencies entry in pnpm-workspace.yaml are removed.

✅ Rule 4 — fixed: rss.xml.ts now busts on publish

src/pages/rss.xml.ts queries D1 and was edge-cached via its routeRules entry but carried no Cache-Tag, so publishing a post never purged it (stale ≤1h). It now calls cache.set({ tags: ["collection:posts"] }) (the cache object on the endpoint's APIContext, the endpoint-side equivalent of Astro.cache) so publish-purge invalidates it. The redundant hand-set Cache-Control header was dropped — the route rule + provider own cache headers.

✅ Rule 3 — fixed: the demo plugin is removed

packages/plugins/demo shipped a sandbox-entry.ts (SandboxedPlugin). Registered in the plugins: array it ran in-process (no Dynamic Worker), so it wasn't an active breach — but it was demo cruft carrying a sandbox adapter that had no place in a hardened starter and could tempt a future move to sandboxed:. It has been deleted: the package, its @local/plugin-demo dependency, its astro.config.mjs import, and its entry in the plugins array are all gone. The production build now ships only first-party plugins plus the local email-resend provider (format: "native", in-process — the model for any new plugin).

Working in this repo

Before committing, run the checklist from CLAUDE.md:

- Touched node_modules / @emdash-cms / patches/?   → stop, use a hook
- New public page?   → routeRules entry + Astro.cache.set(cacheHint)
- New plugin?        → local or @emdash-cms only, in-process, never sandboxed
- New content query? → can it be cached or avoided?
