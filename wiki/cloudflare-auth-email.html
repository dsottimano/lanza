Admin auth & email on Cloudflare (Access, invites, Resend)

Securing the EmDash admin with Cloudflare Access is the clean way to deploy on Workers — one sign-on, no passwords in the CMS. But Access is all-or-nothing: turning it on makes EmDash drop its entire built-in auth surface. Two things you took for granted then stop working and must be wired back: sending email and inviting users.

The one idea: setting auth: access({…}) flips EmDash into external auth mode. In that mode it does not inject its built-in auth routes — passkey, invite, magic-link, recovery all vanish — because identity now comes from the Access JWT. You get seamless SSO; you lose the invite/passkey UI. Plan for it.

1. Access cannot gate *.workers.dev

Access applications only protect hostnames that belong to a zone on your account. A *.workers.dev URL is Cloudflare's shared domain, so Access never sits in front of it — the request reaches the Worker with no JWT and admin hard-fails (a bare 401 from EmDash, no Access challenge).

Fix: give the Worker a custom domain (a real zone you control) and point the Access app at that hostname.

// wrangler.jsonc — route the Worker to a custom domain
"routes": [{ "pattern": "cms.example.com", "custom_domain": true }],
// keep workers.dev alive for staging if you like (admin there just 401s — safe):
"workers_dev": true

Then create the Access application on cms.example.com, copy its AUD tag and team domain, and wire them at build time:

auth = access({
  teamDomain: process.env.CF_ACCESS_TEAM_DOMAIN, // <team>.cloudflareaccess.com
  audience:   process.env.CF_ACCESS_AUD,         // Application Audience (AUD) tag
  autoProvision: true,                           // create a user row on first login
});
How login works now: Access validates the visitor, then EmDash matches/creates a users row by email and mints the session. New users land at the defaultRole (Editor/Author). The built-in EmDash sign-in screen is gone in prod. In dev external auth is bypassed — you fall back to the dev login.

2. Email: register an exclusive provider plugin

EmDash routes every outbound message (invites, magic-link, recovery) through one exclusive email:deliver provider. With no provider, those features fail silently — only a dev-console logger ships built in. A provider is just a trusted in-process plugin (listed under plugins:, not sandboxed:), so it runs in the host Worker with no sandbox and no paid Dynamic Workers.

Why Resend, not Cloudflare Email Sending: CF Email Sending requires the Workers Paid plan. Resend has a free tier and is a plain HTTP API. The starter ships @local/plugin-email-resend; swapping providers is a one-file change in its runtime.ts.

The provider runtime hides five non-obvious traps. Get any wrong and the Worker 500s or email quietly never sends:

  1. Never import a value from "emdash" in the plugin runtime. EmDash calls your createPlugin() at module-eval time while it is still initializing; a value import (even definePlugin) creates a circular dependency → a Cannot access 'X' before initialization TDZ that 500s the whole Worker. Return the resolved-plugin object literally instead. (Types are fine — they erase.)
  2. Capability must be hooks.email-transport:register (the canonical gate for email:deliver). Without it, EmDash silently skips the hook.
  3. The hook must be exclusive: true — EmDash auto-selects the sole exclusive provider; otherwise email.isAvailable() stays false.
  4. Read env inside the handler. import { env } from "cloudflare:workers" forbids property access at module/global scope — touch env.RESEND_API_KEY only inside deliver().
  5. Use format: "native" in the descriptor, not "standard" (standard runs the TDZ-prone adaptSandboxEntry).
// packages/plugins/email-resend/src/runtime.ts — the shape that survives
import { env } from "cloudflare:workers";
import type { EmailDeliverEvent, PluginContext } from "emdash"; // type-only!

async function deliver(event, ctx) {
  const e = env as Record<string, unknown>;          // read INSIDE the handler
  const apiKey = e.RESEND_API_KEY, from = e.EMAIL_FROM;
  const res = await fetch("https://api.resend.com/emails", {
    method: "POST",
    headers: { Authorization: `Bearer ${apiKey}`, "Content-Type": "application/json" },
    body: JSON.stringify({ from, to: event.message.to, subject: event.message.subject,
                           text: event.message.text, html: event.message.html ?? event.message.text }),
  });
  if (!res.ok) throw new Error(`Resend ${res.status}: ${await res.text()}`);
}

export function createPlugin() {
  return {                                            // LITERAL object — no definePlugin()
    id: "email-resend", version: "1.0.0",
    capabilities: ["hooks.email-transport:register"], // the gate
    allowedHosts: [], storage: {}, routes: {}, admin: {},
    hooks: { "email:deliver": { priority: 100, timeout: 5000, dependencies: [],
      errorPolicy: "abort", exclusive: true, handler: deliver, pluginId: "email-resend" } },
  };
}

Then it's gated to Cloudflare in astro.config.mjs (the runtime needs the cloudflare:workers env), and configured with a secret + a var:

const plugins = [auditLog];
if (isCf) plugins.push(emailResendPlugin());

# wrangler secret put RESEND_API_KEY        # your Resend key
# EMAIL_FROM = "My Site <noreply@mail.example.com>"   (a Worker var)
# + verify that sending domain in Resend (adds DKIM/SPF/MX DNS records)

3. Bring back the Invite button

Because external auth drops the built-in auth routes, the admin's Invite User button POSTs to a route that no longer exists — it 404s and the modal hangs on "Sending…". There is also no "create user" endpoint, so otherwise there's no UI way to onboard someone at a chosen role (auto-provisioned logins default to Editor/Author).

Under Access there is no passkey to register, so an "invite" just means create (or re-role) the user row at the chosen role and optionally email them a sign-in link. Re-provide the endpoint yourself:

Astro gotcha: you cannot drop the handler in src/pages/_emdash/api/auth/invite.ts — Astro silently ignores any src/pages path starting with _. Put it outside src/pages and injectRoute it from a tiny integration (the same mechanism EmDash uses). 
// astro.config.mjs — only when external auth is on
const inviteOverride = {
  name: "emdash-invite-override",
  hooks: {
    "astro:config:setup": ({ injectRoute }) => {
      injectRoute({
        pattern: "/_emdash/api/auth/invite",
        entrypoint: fileURLToPath(new URL("./src/server/emdash-invite.ts", import.meta.url)),
      });
    },
  },
};
// integrations: [ react(), ...(auth ? [inviteOverride] : []), emdash({ … }) ]

The handler runs through EmDash's auth middleware like any /_emdash route, so locals.user is the authenticated admin and locals.emdash is the runtime. It validates the caller is an admin, then inserts/updates the users row (id = a ULID, email_verified: 1, the chosen role) and sends a best-effort sign-in email via locals.emdash.email. The invitee then signs in through Access and is matched to the row. See src/server/emdash-invite.ts.

Operational gotchas

  • The plugin-runtime TDZ has a second cause. EmDash 0.19 splits its runtime across chunks; the Cloudflare build can evaluate the plugin-registration array before a const it depends on initializes. The starter collapses emdash into one chunk via vite.build.rollupOptions.output.manualChunks. That covers emdash's own chunks — it does not cover your @local/* plugin chunks, which is the other reason the "return a literal object, no value imports" rule (§2.1) matters.
  • wrangler email … needs the Email Service permission. A delegated/OAuth token without it returns Unauthorized [code: 2036] — onboard the sending domain from the dashboard instead. (Moot if you use Resend: domain verification happens in Resend, not Cloudflare.)
  • Never run pnpm build from inside node_modules/emdash. Its own build script is tsdown; you'll get tsdown: not found and think your app broke. Always build from the app root.
  • If you ever turn external auth off, delete src/server/emdash-invite.ts and its injectRoute entry — EmDash will re-inject its real invite route and the two would collide.
