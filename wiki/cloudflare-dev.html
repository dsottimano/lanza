Cloudflare dev & the reload loop

When you move an EmDash site from the node adapter to Cloudflare, pnpm dev stops reading data.db and starts running on miniflare — and it will drop into an endless [vite] program reload loop on every request until you tell Vite to stop watching one directory.

The one idea: the Cloudflare adapter persists D1/R2/KV to .wrangler/state/ inside your project. EmDash writes to D1 on every request, so those files churn constantly. Add .wrangler to vite.server.watch.ignored or dev is unusable.

What changes in dev when you go Cloudflare

On the node adapter the database is a plain file the server opens directly:

// node adapter
adapter: node({ mode: "standalone" }),
database: sqlite({ url: "file:./data.db" }),
storage:  local({ directory: "./uploads" }),

On the Cloudflare adapter the database and storage become bindings, and astro dev emulates them locally with miniflare:

// cloudflare adapter
adapter: cloudflare(),
database: d1({ binding: "DB" }),      // from @emdash-cms/cloudflare
storage:  r2({ binding: "MEDIA" }),
  • Dev now reads/writes local D1/R2/KV under .wrangler/state/v3/ — not your old data.db and not the remote Cloudflare resources.
  • That local D1 starts empty, so on first request EmDash runs its migrations and seeds the demo content. You will not see your real site until you load data into the local D1 (see last section).

The bug: [vite] program reload on every request

EmDash touches D1 on ordinary page loads — sessions, rate-limit counters, the 404 log, view tracking, and so on. miniflare flushes each write to disk:

.wrangler/state/v3/d1/miniflare-D1DatabaseObject/<hash>.sqlite-wal
.wrangler/state/v3/d1/miniflare-D1DatabaseObject/<hash>.sqlite-shm

Vite watches the whole project tree. It sees those *.sqlite-wal writes, fires a full program reload, the worker re-initializes, that touches D1 again, which writes state again — and the loop never settles. A single request can spawn a dozen reloads.

How it was pinned down: with the server idle, no reloads. Send one request and find . -newer marker -not -path './node_modules/*' reports only the two .sqlite-wal/-shm files — nothing in src/. The watcher, not your code, is the trigger.

The fix

Exclude the runtime-state and build directories from Vite's file watcher in astro.config.mjs:

vite: {
  plugins: [tailwindcss()],
  optimizeDeps: { include: ["@phosphor-icons/react"] },
  server: {
    watch: {
      ignored: ["**/.wrangler/**", "**/data.db*", "**/dist/**"],
    },
  },
},

After this, requests no longer trigger reloads (verified: six requests, six 200s, zero reloads). The entry is harmless on the node adapter, so it is safe to keep in a starter that hasn't switched to Cloudflare yet.

Bonus: making dev show your real content

Because local D1/R2 start empty, mirror your content into them so dev matches production:

# 1. load a SQL dump into the LOCAL D1 (wipe the demo seed first)
rm -rf .wrangler/state/v3/d1
wrangler d1 execute <db-name> --local --file d1-import.sql --yes

# 2. push every media file into the LOCAL R2 bucket (object key = storage_key)
for f in uploads/*; do
  wrangler r2 object put "<bucket>/$(basename "$f")" --file "$f" --local
done

One caveat exporting SQLite → D1: a plain .dump fails on D1 because D1 rejects forward foreign-key references (a table that references users created before users exists). Re-emit the dump with tables in dependency order, load the data, then rebuild any FTS5 indexes with INSERT INTO <fts_table>(<fts_table>) VALUES('rebuild').
