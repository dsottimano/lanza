Building a wiki inside EmDash

Three viable routes to put a wiki in /admin, ranked by effort vs. authoring richness. (This static HTML wiki is a fourth, simplest option — outside EmDash entirely.)

OptionEffortAuthoring richnessBest for
A. CollectionLowestHigh (Portable Text)An internal knowledge base
B. Native React pluginHighAnythingA bespoke admin app
C. Block Kit pluginMediumLowA portable/marketplace tool

Option A — a docs/wiki collection (recommended)

No plugin. Define a collection with title + a Portable Text content field. You instantly get a "Wiki" sidebar item, the full rich-text editor (headings, lists, code, links, images, tables), built-in search, drafts, and multi-user editing.

{
  "name": "wiki",
  "label": "Wiki",
  "labelSingular": "Article",
  "fields": [
    { "name": "title",   "type": "text" },
    { "name": "content", "type": "portableText" }
  ]
}

Caveat: a collection is "content," so it's queryable by the public site. For an internal wiki, simply don't build public routes for it (or gate access) — it stays admin-only.

Option B — native plugin with a React admin page

Native plugins (declared in astro.config.mjs, trusted mode — which is how your plugins already run) can ship React components mounted in /admin via an adminEntry / ./admin export. That's a fully custom wiki UI: your own layout, tree navigation, markdown rendering, direct storage access. Most flexible, most work. Only worth it if Option A's "it's just a content collection" framing is a dealbreaker.

Option C — standard plugin with a Block Kit admin page

What the demo plugin does. adminPages adds the sidebar item; the handler returns Block Kit blocks and reacts to interactions, with a plugin-private storage collection for the articles:

  • list view → a table of articles,
  • read view → section / code / banner blocks,
  • edit → a form with a multiline text_input,
  • navigation → button actions that re-render different blocks.

Limits: no rich-text editor and no freeform HTML (only Block Kit primitives); every interaction is a server round-trip (no client JS in sandbox mode); "pages" are interaction-state, not bookmarkable URLs. Functional but a clunky authoring experience. Its only edge is sandbox/marketplace portability — which an internal first-party wiki doesn't need.

Verdict

For an internal EmDash wiki, model a wiki collection (Option A) — you're writing in the real editor within minutes, with zero plugin code. Reach for B only when you need a bespoke UI, and C only when you need marketplace portability.
See Plugin capabilities & limits and Block Kit for the constraints behind options B and C · EmDash 0.16.1.
