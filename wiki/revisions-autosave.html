Revisions, autosave & discarding drafts

Why the inline editor pile up dozens of revisions you don't remember making, why they show no author, and how to fix both with a single Astro middleware. No core EmDash code was modified.

The one idea: core already supports coalescing autosaves (skipRevision) and authored revisions — the inline visual editor just doesn't use them. The fix is to fill in what the inline editor omits, in front of the API, via middleware.

The revision data model

For collections declaring supports: ["revisions"], EmDash splits content storage in two:

  • Published data lives in the content table columns (ec_pages, ec_posts, …). These are never written by ordinary saves.
  • Draft data lives only in the revisions table. The content row points at its current draft via draft_revision_id.
  • Publishing promotes the draft into the columns and clears draft_revision_id (see publish in content.ts).

So a page with unpublished edits has a non-null draft_revision_id; a clean published page has draft_revision_id = NULL and its real content sitting safely in the columns.

The bug: a revision per field-blur, with no author

The inline visual-editing toolbar (the floating Edit · Publish bar on the public site) saves on every field blur — click out of a field, or press Enter:

// emdash/src/visual-editing/toolbar.ts  (saveField)
ecFetch("/_emdash/api/content/" + collection + "/" + id, {
  method: "PUT",
  body: JSON.stringify({ data: { [field]: value } }),   // ← no skipRevision, no authorId
});

Core's update path (emdash-runtime.ts) only coalesces when the request asks it to:

if (body.skipRevision && existing.draftRevisionId) {
  await revisionRepo.updateData(existing.draftRevisionId, mergedData);   // update in place
} else {
  await revisionRepo.create({ ..., authorId: body.authorId ?? undefined });   // NEW row
}

The inline toolbar never sends skipRevision, so every blur inserts a brand-new revision row. A single editing session balloons into dozens. And because it never sends authorId (and the route doesn't fall back to the session user for the revision author), every one of them is stored with author_id = NULL.

The full admin dashboard editor (/_emdash/admin) does not have this problem — it sends skipRevision: true on autosave. Only the inline toolbar is affected.

The fix: one middleware, no core changes

The only app-level seam in front of that PUT is Astro middleware. We default the flag the inline editor omits, and backfill the author from the session — only when the key is absent, so the admin editor's explicit choice is never overridden:

// src/middleware.ts
import { defineMiddleware } from "astro:middleware";

const CONTENT_ITEM_PUT = /^\/_emdash\/api\/content\/[^/]+\/[^/]+$/;

export const onRequest = defineMiddleware(async (context, next) => {
  const { request, locals } = context;
  if (request.method === "PUT" && CONTENT_ITEM_PUT.test(new URL(request.url).pathname)) {
    try {
      const body = await request.clone().json();
      if (body && typeof body === "object") {
        let changed = false;
        if (body.skipRevision === undefined) { body.skipRevision = true; changed = true; }
        const userId = locals.user?.id;
        if (body.authorId === undefined && userId) { body.authorId = userId; changed = true; }
        if (changed) {
          context.request = new Request(request.url, {
            method: request.method, headers: request.headers, body: JSON.stringify(body),
          });
        }
      }
    } catch { /* non-JSON body: leave untouched */ }
  }
  return next();
});

Two things make this safe and precise:

  • Absent-only injection. The admin editor always sets skipRevision explicitly; the inline editor never does. === undefined targets exactly the inline editor.
  • Ordering. EmDash registers its auth middleware with order: "pre", so locals.user is already populated when ours runs — the authorId backfill works.

First save of a session still creates one revision (core only coalesces once a draft exists), so you get exactly one draft revision per session — not zero, not 18.

Verification

Three inline-style PUTs (no skipRevision in the body) against an isolated published page, with and without the middleware:

ScenarioNew revisions from 3 savesRevision author
Control — no middleware3 (appends each save)null
With middleware1 (coalesced in place)session user

The rewritten body reaches the route uncorrupted (final draft data was exactly the last edit), and astro check stays clean.

How to discard a draft

The inline toolbar has no discard button — only Publish. The "Discard changes" button lives in the admin dashboard editor. Three ways to drop a pending draft and revert to the published version:

  1. Admin dashboard: /_emdash/admin → open the item → Discard changes → confirm.
  2. API: POST /_emdash/api/content/{collection}/{id}/discard-draft (authenticated). It nulls draft_revision_id; published columns already hold the live version.
  3. Direct DB (what both do under the hood): UPDATE ec_pages SET draft_revision_id = NULL WHERE id = '…';
The revert ↶ arrows in the Revisions panel are not discard — each one writes a new revision rolling content back, which is part of why the pile grows. Discard drops the draft pointer without adding history.

Pruning accumulated revisions

Core already caps history at 50 per entry (pruneOldRevisions), so growth is bounded but can still be cluttered. To hard-reset a page that only has draft-iteration noise (no meaningful published checkpoints — live_revision_id is null and published data is in the columns), back up first, then:

-- after discarding the draft pointer
DELETE FROM revisions WHERE entry_id = '<id>';   -- safe: published data is in ec_pages columns
Caveat — not retroactive. The middleware only changes behavior going forward. Existing null-author revisions stay null; the inline Revisions panel may not show authors for new ones until a reload, even though the DB attribution is correct.

Upstream

Both root causes are reported to the official EmDash repo — this middleware is a workaround until they land:

  • Issue: inline visual editor omits skipRevision, creating a revision per field-blur.
  • Issue: revision author not derived from the session (visual-editor saves store author_id = NULL).
  • Discussion: revert-appends — reverting writes a new revision rather than moving the draft pointer.
